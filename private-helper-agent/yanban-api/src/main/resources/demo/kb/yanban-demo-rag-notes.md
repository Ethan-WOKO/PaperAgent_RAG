# RAG 流程与评测笔记

这篇文档用于 Demo 知识库问答，描述研伴 ScholarAI 当前的 RAG 链路和下一阶段 Agent 体验改进计划。

## 从文档上传到生成回答的步骤

1. 文档上传：用户在知识库页面上传 Markdown、TXT、PDF 或 DOCX，系统记录文件名、大小、类型、来源和处理状态。
2. 文本解析：后端抽取正文内容。普通文本直接读取；PDF、DOCX 等文件先转换为可检索文本。
3. 文档切块：正文按窗口切成知识片段，默认每个 chunk 约 500 个字符，并保留文档 ID、用户 ID、来源类型和顺序信息。
4. 生成 embedding：每个 chunk 调用向量模型生成 embedding，便于后续语义召回。
5. 写入索引：chunk 文本、元数据和向量写入 Elasticsearch，同时数据库保存文档和 chunk 状态。
6. 问题理解：用户提问后，系统判断是否需要知识库、网页搜索或其他工具；普通常识问题可以直接回答。
7. 检索召回与重排：系统从知识库召回候选片段，再结合关键词命中、向量分数和重排结果选择证据。
8. 生成回答：模型根据用户问题、最近对话和检索证据生成最终回答；如果证据不足，应明确说明不确定性。

## 推荐评测指标

- Recall@5：正确证据是否出现在前 5 个召回片段里。
- MRR：第一个正确证据出现得越靠前越好。
- Faithfulness：回答中的结论是否能被检索证据支持。
- Citation Coverage：关键结论是否覆盖到了对应来源。
- Latency：一次问答从发送到首 token、从发送到完整回答分别耗时多久。

## 两周内改进 Agent 体验的优先任务

第一优先级是会话稳定性：发送中切换页面或切换会话时，不应丢失用户消息；WebSocket 失败时要自动降级到普通 HTTP 发送。

第二优先级是工具调用质量：工具描述要包含触发条件、输入格式、输出格式和失败处理方式；模型输出工具调用前要做格式校验，避免伪工具调用文本直接展示给用户。

第三优先级是检索策略：不要让所有问题都搜索网页，要由模型或策略判断是否需要工具；对常识问题优先直接回答，对时效性、事实核验和私有资料问题再调用搜索或知识库。

第四优先级是 Demo 体验：示例问题要能命中预置知识库，移动端不能横向溢出，游客账号需要定时重置并限制上传和配置修改。
